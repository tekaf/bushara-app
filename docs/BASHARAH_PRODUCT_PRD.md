Basharah – Official Product & System Scenario Reference (V1)
1) Purpose of this document

هذا الملف هو المرجع الرسمي الأساسي لمشروع Basharah.
الغرض منه توضيح منطق المنصة، رحلة المستخدم، حالات النظام، قواعد العمل، والأنظمة الفرعية المرتبطة بالدعوات الإلكترونية.

Important rule

أي تعديل جديد أو ميزة جديدة في المشروع يجب:

مراجعتها على هذا المرجع أولًا

تحديث المرجع إذا لزم

ثم تنفيذها برمجيًا

هذا الملف ليس مجرد وصف عام، بل هو Product + Business Logic + System Behavior Reference.

2) Product overview

Basharah منصة إلكترونية لإنشاء وإدارة وإرسال الدعوات الإلكترونية للمناسبات عبر واتساب فقط.

المنصة تسمح للمستخدم بـ:

اختيار باقة حسب عدد المدعوين

اختيار نوع المناسبة

اختيار تصميم الدعوة

إدخال بيانات المناسبة

الدفع

انتظار مراجعة الدعوة من الإدارة

استلام Preview بعد الدفع والمراجعة

إضافة المدعوين

جدولة إرسال الدعوات عبر واتساب

متابعة الردود (قبول / اعتذار)

إرسال تذكير لاحق للمقبولين فقط

تقسيم الباقة بين أطراف متعددة مثل أهل العريس / أهل العروس

3) Core business model
3.1 الباقات

النظام يعتمد على باقات بعدد المدعوين، مثل:

75 مدعو

الإرسال يكون عبر:

WhatsApp only

لا يوجد SMS في هذه المرحلة.

أي أن:

المستخدم يدفع أولًا

ثم تدخل الدعوة مرحلة المراجعة

ثم بعد اعتمادها يمكن عرض Preview للمستخدم

7) What happens after payment

بعد نجاح الدفع، يقوم النظام تلقائيًا بـ:

إنشاء Order

إنشاء Invoice

إنشاء Invitation Record

ربط الدعوة بالمستخدم

ربط الدعوة بالباقة المختارة

وضع الدعوة في حالة Pending Review
بعد الدفع تدخل الدعوة إلى ورشة التأكد

ويرسل النظام بريدًا إلكترونيًا للإدارة

البريد يحتوي رابط دخول مباشر لصفحة المراجعة

قاعدة تشغيل إلزامية:

إشعار ورشة التأكد للإدارة (Workshop Review Email) إلزامي بعد كل عملية دفع ناجحة.

إذا فشل إرسال البريد لأي سبب، يجب تسجيل حالة واضحة داخل الدعوة مثل `adminNotificationStatus = pending/failed`

ويجب عدم فقدان الطلب أو إخفاء الحالة عن الفريق الداخلي.

المستخدم لا يرى الـ Preview قبل انتهاء هذه المرحلة

تحويلها إلى لوحة الإدارة للمراجعة

بعد الدفع تظهر للمستخدم:

صفحة الفاتورة

رقم الطلب

تاريخ الطلب

الباقة

المبلغ

حالة الدفع

حالة الدعوة الحالية

قاعدة تجربة المستخدم بعد الدفع:

لا تظهر للمستخدم أي أزرار تشجعه على تعديل محتوى الدعوة بعد الدفع.

الأزرار المسموح بها في شاشة حالة الطلب تكون للعودة إلى لوحة المستخدم / إدارة الدعوات / الباقات فقط.

مثال على حالة الدعوة بعد الدفع

Payment Status: Paid

Invitation Status: Pending Review

11) Admin actions

11.2 Contact Customer / التواصل مع العميل

11.3 Return for correction

12) Invitation statuses

يجب أن يكون لكل دعوة حالة واضحة.
هذا جزء أساسي جدًا في النظام.
أضف حالة واضحة تعبر عن “ورشة التأكد”.

مثلًا بين:

paid

و approved

يجب أن يكون فيه معنى مثل:

in_workshop_review
أو

awaiting_admin_confirmation

الحالات المقترحة:

draft

awaiting_payment

paid

pending_review

needs_customer_update

approved

ready_for_scheduling

scheduled

sending

partially_sent

sent

completed

cancelled

شرح الحالات

draft: الدعوة تحت الإنشاء قبل الدفع

awaiting_payment: المستخدم أكمل الخطوات ولم يدفع بعد

paid: الدفع تم بنجاح

pending_review: تنتظر مراجعة الإدارة

needs_customer_update: تحتاج تعديل من العميل

approved: تم اعتمادها

ready_for_scheduling: معتمدة وجاهزة إذا اكتمل شرط المدعوين

scheduled: تم تحديد وقت الإرسال وتجهيزها للجدولة

sending: جاري إرسال الرسائل

partially_sent: تم إرسال جزء وفشل جزء

sent: تم تنفيذ الإرسال

completed: انتهت الرحلة الأساسية وبدأت الردود والتذكيرات

cancelled: ألغيت

26) Shared package / family split system

يمكن للمستخدم الأساسي تقسيم جزء من باقته لطرف آخر.

مثال

باقة 100:

60 للمستخدم الأساسي

20 لأم العريس

20 لأم العروس

النظام ينشئ لكل حصة:

Record خاص

رابط خاص

عدد مخصص

مالك خاص

الطرف الثانوي عند دخوله:

يسجل الدخول برقم الجوال

يدخل فقط على حصته

يضيف مدعويه فقط

لا يرى الآخرين

قاعدة أساسية

لا يتم دمج بيانات الحضور والاعتذار بصريًا بشكل يسبب لخبطة بين الأطراف داخل واجهة الإدخال.
لكن يمكن للإدارة أو المستخدم الأساسي رؤية التقارير النهائية المجمعة مع فصل المصدر.

27) Ownership separation

كل مدعو يجب أن يكون مربوطًا بـ:

الدعوة الأم

الجهة المالكة لهذا المدعو

مثال:

ownerType = primary_user

ownerType = shared_sub_user

والهدف:

معرفة من أضاف المدعو

عدم خلط الردود

عدم خلط الأعداد

تمكين التقارير الصحيحة

28) Reminder system

المستخدم يستطيع تحديد وقت تذكير عند إعداد الدعوة أو لاحقًا حسب التصميم النهائي للنظام.

قاعدة التذكير

التذكير يُرسل فقط إلى:

الذين قبلوا الدعوة

لا يرسل إلى:

المعتذرين

الذين لم يتم إرسال دعوتهم

المدعوين الفاشلين إلا إذا تم إصلاح حالتهم

التذكير أيضًا يجب أن يكون:

مجدولًا

قابلًا للتتبع

له حالة مستقلة

29) Failed sends and retry logic

إذا فشل إرسال الدعوة إلى مدعو:

يظهر السبب إن أمكن

يمكن إعادة المحاولة

مطلوب دعم:

إعادة إرسال الرسائل الفاشلة

إعادة إرسال دعوة لمدعو واحد

إعادة المحاولة الجماعية للفاشلين

30) Replace guest logic

يجب دعم السيناريو التالي:

حذف مدعو قبل الإرسال

إضافة مدعو جديد بدلًا عنه

القاعدة

يمكن الحذف والاستبدال

38) Edge cases

لا بد أن يغطي النظام الحالات التالية:

دفع ناجح لكن إنشاء الطلب فشل

إنشاء الدعوة تم لكن لم تصل للأدمن

الدعوة معتمدة لكن لا يوجد مدعوون

يوجد مدعوون لكن الموعد لم يحدد

بعض الأرقام غير صحيحة

بعض الرسائل فشلت

المدعو فتح الرابط مرتين

المدعو رد ثم حاول تغيير الرد

المستخدم حذف مدعوًا بعد الجدولة

المستخدم قسم الباقة ثم تجاوز الحد

الطرف الثانوي حاول رؤية بيانات غيره

التذكير تم تحديده قبل موعد الدعوة بشكل غير منطقي

وقت الإرسال صار في الماضي بسبب تعديل أو خطأ زمني

الأدمن تواصل مع العميل لكن لم يعدّل الدعوة

الدعوة تحتاج تحديث بعد الاعتماد

39) Rules that must not be broken

هذه قواعد أساسية للمشروع:

لا إرسال قبل الدفع

لا Preview نهائي قبل الدفع

بعد الدفع توجد مراجعة داخلية

بعد الدفع: الدعوة مقفلة نهائيًا على المستخدم (No Edit by Customer)

لا يسمح للمستخدم بالعودة لشاشة تحرير تفاصيل الدعوة بعد نجاح الدفع

الإرسال عبر واتساب فقط

الرد مرة واحدة فقط

لكل مدعو رابط آمن وفريد

لا يجوز تجاوز حد الباقة

لكل مدعو مالك واضح

لا تخلط ردود الحصص المختلفة داخل واجهة التحكم

يجب تسجيل جميع الحالات المهمة في النظام

فشل الإرسال لا يعني ضياع السجل

يجب دعم إعادة الإرسال

يجب دعم حذف واستبدال المدعو قبل الإرسال

يجب أن يفهم المستخدم حالة دعوته دائمًا

40) Development rule for Cursor

أي ميزة جديدة يتم بناؤها في Basharah يجب أن تراعي:

هل تتوافق مع هذا المرجع؟

هل تحتاج تحديث حالة جديدة؟

هل تحتاج تحديث قاعدة عمل؟

هل تؤثر على الدعوة أو المدعو أو الإرسال أو الردود؟

هل تضيف Edge Cases جديدة؟

هل تحتاج تحديث لوحة المستخدم أو لوحة الأدمن؟

هل تحتاج تحديث هذا الملف قبل التنفيذ؟

سياسة التنفيذ الإلزامية (System-wide impact):

يجب الالتزام بسياسة التنفيذ الرسمية الموجودة في:

`docs/BASHARAH_IMPLEMENTATION_POLICY.md`

ولا يسمح بالتنفيذ الموضعي فقط عند وجود أثر أوسع على النظام.
