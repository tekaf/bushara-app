# BASHARAH Implementation Policy

النسخة: 1.0  
الحالة: Active (إلزامي)

## الهدف

ضمان أن أي تطوير جديد في مشروع Basharah يتم بعقلية **Product/System-wide impact** وليس تعديلًا موضعيًا فقط.

## سياسة التنفيذ الإلزامية

### 1) فهم أثر التعديل على النظام بالكامل

قبل أي تنفيذ، يجب تقييم:
- هل التعديل يؤثر على صفحات أخرى مشابهة؟
- هل يوجد عناصر/تسميات/أزرار يجب توحيدها في النظام؟
- هل يوجد سلوك قديم أصبح غير مناسب ويجب حذفه أو استبداله؟
- هل يوجد تعارض محتمل مع flow أو navigation أو UI consistency؟

### 2) البحث عن الحالات المشابهة

إذا تم إضافة عنصر جديد (زر/تسمية/سلوك):
- لا يُطبق على صفحة واحدة فقط.
- يجب البحث عن جميع الصفحات المشابهة.
- يجب توحيد السلوك والتسمية والتصميم عبر الصفحات ذات العلاقة.

### 3) الحفاظ على الاتساق

يمنع اختلاف المصطلحات بدون سبب معتمد.

مثال:  
لا تكون التسميات بين الصفحات: `رجوع` / `العودة` / `عودة` بشكل عشوائي.

المطلوب: مصطلحات موحدة + نمط بصري متسق.

### 4) مراجعة ما يجب حذفه بعد الإضافة

لا يكفي إضافة الجديد فقط.

بعد كل تعديل يجب مراجعة:
- هل يوجد جزء قديم أصبح غير لازم؟
- هل يوجد UI قديم يجب إزالته؟
- هل يوجد سلوك سابق يجب تعديله ليتماشى مع التحديث؟

### 5) إجراء إلزامي قبل كتابة الكود

قبل التنفيذ في أي Feature جديدة، يجب تقديم ملخص مختصر يشمل:
- الصفحات المتأثرة.
- العناصر التي ستوحد.
- الأجزاء القديمة التي ستُحذف/تُعدل.
- التعارضات أو الآثار الجانبية المحتملة.

ثم يبدأ التنفيذ.

### 6) منع التنفيذ عند خلق عدم اتساق

إذا كان الطلب سيخلق عدم اتساق بالنظام:
- يجب التنبيه أولًا.
- لا يتم التنفيذ الموضعي مباشرة.
